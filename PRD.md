# ContentForge PRD v2

> 开源 AI 驱动的内容中台 · 2026-06-06（需求完善版：2026-06-11）
> 多开发人员可对照此文档独立开发，功能行为保持一致。

---

## 1. 产品一句话

一个能自部署的 Web 应用，让团队用 AI 批量生成营销文案、博客、社媒帖子，经过人工审核后发布——全程在一个平台完成，不用在 ChatGPT 和文档之间来回粘贴。

---

## 2. 用户是谁

小团队的内容运营 or 市场人员（3-20 人），买不起 Jasper（$59/人/月），但每天要产出各种文本内容。

### 他们的痛点

| 现在怎么干的 | 问题 |
|:--|:--|
| 打开 ChatGPT，写 prompt → 复制结果 → 贴到文档 → 改 → 发 | 切换 4 个工具 |
| 每个平台 prompt 不一样，每次都重写 | 没有模板复用 |
| AI 生成的内容有时乱写，没人审核直接发了 | 质量事故 |
| 团队三个人，谁写了什么、发了什么，完全不知道 | 无协作 |

---

## 3. 解决什么问题

**一句话：把 AI 内容生产从"手工小作坊"变成"工厂流水线"。**

```
没有 ContentForge：
  运营小妹 → ChatGPT → 复制 → 粘贴到微博 → 再复制 → 粘贴到公众号 → 忘了审核直接发 → 被老板骂

有了 ContentForge：
  运营小妹 → 选模板 → 填两个词 → 点生成 → 组长审核 → 一键发布 → 下班
```

## 4. 核心功能

### 4.1 用户认证

| 功能 | 说明 |
|:--|:--|
| 邮箱注册 | 输入邮箱 + 密码 + 姓名，创建账号（默认 EDITOR 角色） |
| 邮箱登录 | 邮箱 + 密码登录，支持「记住我」 |
| 会话管理 | JWT 策略，无需数据库 session 表；登录后 30 天有效 |
| 路由保护 | 未登录访问 `/dashboard/*` `/projects/*` 自动重定向到 `/login` |
| 角色区分 | ADMIN 可创建项目；EDITOR 只能在自己已加入的项目空间内操作 |

**认证架构**

```
用户 → 登录页(/login) → 输入邮箱+密码 → NextAuth Credentials Provider
                                              │
                           ┌──────────────────┘
                           ▼
                      Prisma 查 User 表
                      (对比 bcrypt hash)
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
          密码错误       用户不存在    验证通过
          返回错误      返回错误     签发 JWT
                                       │
                          ┌────────────┘
                          ▼
              SessionProvider 包裹全应用
              (useSession 读取当前用户)
```

**安全要求**
- 密码使用 bcrypt 哈希存储，不存明文
- NEXTAUTH_SECRET 通过环境变量注入
- 登录次数不做限制（MVP 阶段不引入 rate limit）
- 密码最小长度 8 位

### 4.2 模板管理

模板 = 一个可复用的 prompt 骨架，含 `{{变量名}}` 占位符。运营人员创建模板后，其他成员用它一键生成内容。

#### 4.2.1 模板字段

| 字段 | 类型 | 必填 | 说明 |
|:--|:--|:--|:--|
| 名称 | 文本 | ✅ | 模板显示名，如"小红书种草文案" |
| 分类 | 下拉选择 | ✅ | BLOG / SOCIAL / AD / EMAIL / CUSTOM |
| Prompt 模板 | 多行文本 | ✅ | 含 `{{变量名}}` 占位符的 prompt 正文 |
| 变量列表 | 自动提取 | — | 从 prompt 模板中扫描 `{{...}}` 自动生成，只读展示 |

**变量自动提取规则**：解析 prompt 模板中所有 `{{变量名}}`，去重后生成变量列表。变量名只允许中文/字母/数字，不允许嵌套 `{{}}`。

#### 4.2.2 操作

| 操作 | 谁可以做 | 行为 |
|:--|:--|:--|
| 查看模板列表 | 项目成员 | 按分类筛选，卡片或表格展示 |
| 创建模板 | 项目成员 | 填写表单 → 保存 → 回到列表，新模板出现在顶部 |
| 编辑模板 | 创建者本人 | 修改字段 → 保存 → 回到列表 |
| 删除模板 | 创建者本人 | 弹窗确认 → 删除（已有内容引用该模板的，templateId 置空，不删内容） |

#### 4.2.3 举个例子

创建"小红书种草文案"模板，填入：

```
名称：小红书种草文案
分类：SOCIAL
Prompt 模板：
你是一个小红书种草博主，请为「{{产品名}}」写一篇种草文案。

要求：
- 目标用户：{{目标人群}}
- 风格：{{风格}}
- 字数：{{字数}}字以内
- 包含 3-5 个 emoji
```

系统自动提取变量：`[产品名, 目标人群, 风格, 字数]`。

#### 4.2.4 状态

| 状态 | 何时出现 | 展示 |
|:--|:--|:--|
| 空列表 | 项目下尚无模板 | 插图 + "还没有模板，创建第一个吧" + 创建按钮 |
| 加载中 | 数据请求中 | Skeleton 卡片 × 3 |
| 加载失败 | 网络/服务端错误 | "加载失败" + 重试按钮 |

### 4.3 AI 内容生成

核心流程：**选模板 → 填变量 → AI 生成（流式） → 编辑确认 → 保存为草稿**。

#### 4.3.1 生成流程（分步）

**Step 1：选模板**
- 下拉列表展示当前项目所有模板，按名称排序
- 选中模板后，下方展示该模板的变量列表

**Step 2：填写变量**
- 每个变量对应一个文本输入框，label 为变量名
- 必填校验：所有变量不能为空
- 下方展示完整的 prompt 预览（变量替换后的实际 prompt），实时更新

**Step 3：点击「生成」**
- 按钮变为 loading 状态（spinner + "生成中..."），禁止重复点击
- 调用 DeepSeek API，SSE 流式返回
- 正文区域以打字机效果实时展示生成内容

**Step 4：结果操作**
- 生成完成后，正文区域变为可编辑（富文本或纯文本 textarea）
- 用户可手动修改正文
- 提供三个按钮：「重新生成」「保存草稿」「提交审核」

#### 4.3.2 操作

| 操作 | 行为 |
|:--|:--|
| 生成 | 调 DeepSeek API → SSE 流式展示 |
| 停止生成 | 生成中显示「停止」按钮 → 中断 SSE 连接，保留已生成文本 |
| 重新生成 | 清空正文 → 重新调 API（不回到填变量步骤） |
| 保存草稿 | 保存当前正文 + 元信息 → 状态 = DRAFT → 跳转到内容详情页 |
| 提交审核 | 保存当前正文 → 状态 = PENDING → 跳转到内容详情页 |
| 手动编辑 | 直接在正文区修改，自动保存到本地 state（不自动写库） |

#### 4.3.3 状态

| 状态 | 何时出现 | 展示 |
|:--|:--|:--|
| 无模板 | 项目下没有模板 | "请先创建模板" + 跳转到模板列表的链接 |
| 生成中 | API 调用进行中 | 正文区打字机效果 + "停止"按钮 + 上方变量区禁用 |
| 生成完成 | 流式结束 | 正文区可编辑 + 三个操作按钮 |
| 生成失败 | API 报错/超时 | Toast 错误提示 + "重新生成"按钮恢复可用 |

### 4.4 内容审核与状态流转

```
草稿 ──→ 待审核 ──→ 已通过 ──→ 已发布
  ↑         ↓
  └── 已驳回（附驳回理由）
```

#### 4.4.1 状态定义与权限矩阵

| 状态 | 含义 | 谁可以看 | 谁可以编辑正文 | 可用操作 |
|:--|:--|:--|:--|:--|
| DRAFT 草稿 | 刚生成/正在编辑 | 作者本人 | ✅ 作者 | 编辑 → 保存 / 提交审核 / 删除 |
| PENDING 待审核 | 已提交，等待审核 | 项目所有成员 | ❌ | 通过 / 驳回（编辑=作者，审核=同项目任何人） |
| APPROVED 已通过 | 审核通过，可发布 | 项目所有成员 | ❌ | 发布 / 驳回 |
| REJECTED 已驳回 | 审核不通过，退回修改 | 作者本人 | ✅ 作者 | 编辑 → 保存 / 提交审核 / 删除 |
| PUBLISHED 已发布 | 已标记发布 | 项目所有成员 | ❌ | 无（终态） |

#### 4.4.2 审核操作

| 操作 | 触发条件 | 行为 |
|:--|:--|:--|
| 提交审核 | 状态 = DRAFT 或 REJECTED，作者本人 | 状态 → PENDING，记录 authorId |
| 通过 | 状态 = PENDING，审核人 ≠ 作者 | 状态 → APPROVED，记录 reviewerId |
| 驳回 | 状态 = PENDING 或 APPROVED，审核人 ≠ 作者 | 弹出驳回理由输入框（必填，最多 500 字）→ 状态 → REJECTED，记录 reviewerId + rejectionReason |
| 发布 | 状态 = APPROVED，作者本人 | 状态 → PUBLISHED |

#### 4.4.3 特殊规则
- 同一个人不能审核自己提交的内容（审核按钮对作者本人隐藏）
- 驳回理由为必填，不能为空
- 已发布为终态，不可再修改

### 4.5 仪表盘

登录后着陆页。跨项目聚合用户的所有内容概况。

#### 4.5.1 展示数据

| 区块 | 数据 | 计算方式 |
|:--|:--|:--|
| 本周生成数 | 数字 + "篇" | 本周一 0:00 至今，该用户创建的 Content 数量 |
| 待审核数 | 数字 + "篇" | 全平台状态 = PENDING 的 Content 数量 |
| 状态分布 | 饼图或横向条形图 | DRAFT / PENDING / APPROVED / REJECTED / PUBLISHED 各多少 |
| 最近内容 | 列表（最新 5 条） | 按 updatedAt 倒序，跨项目混合展示，每条显示：标题 / 状态标签 / 所属项目名 / 更新时间 |

#### 4.5.2 交互
- 点击「最近内容」列表项 → 跳转到该内容的详情页
- 点击「待审核数」→ 跳转 `/projects/[id]/contents?status=PENDING`（若有多个项目，跳到第一个有 PENDING 的项目）
- 页面顶部显示用户名 + "下午好/早上好"（根据当地时间）

#### 4.5.3 状态
| 状态 | 展示 |
|:--|:--|
| 新用户（0 内容） | 数字全为 0，最近内容区显示 "还没有内容，去创建第一篇吧" + 跳转按钮 |
| 加载中 | Skeleton 卡片 |
| 加载失败 | "加载失败" + 重试 |

### 4.6 项目空间

项目 = 内容 + 模板的隔离容器。用户可以创建/加入多个项目。

#### 4.6.1 项目字段

| 字段 | 类型 | 必填 | 说明 |
|:--|:--|:--|:--|
| 名称 | 文本 | ✅ | 项目名，如"618 大促" |
| 描述 | 多行文本 | 否 | 一句话说明项目用途 |

#### 4.6.2 操作

| 操作 | 谁可以做 | 行为 |
|:--|:--|:--|
| 查看我的项目列表 | 所有登录用户 | 展示已加入的所有项目，卡片式布局 |
| 创建项目 | 所有登录用户 | 填写名称+描述 → 创建者自动成为 owner（ADMIN 角色）+ 成员 |
| 进入项目 | 项目成员 | 点击项目卡片 → 跳转 `/projects/[id]`（项目详情页） |
| 加入项目 | 所有登录用户 | 在项目列表页有"加入项目"入口 → 输入项目 ID → 自动成为成员（EDITOR 角色） |
| 查看项目成员 | 项目成员 | 项目详情页展示成员列表（姓名 + 角色） |

#### 4.6.3 项目详情页（`/projects/[id]`）

页面布局：
- 顶部：项目名称 + 描述 + 成员头像列表
- 两个 Tab：「内容」和「模板」
- 内容 Tab：内容列表（见 4.4），带状态筛选下拉框
- 模板 Tab：模板列表（见 4.2），带分类筛选下拉框
- 右下角 FAB（浮动操作按钮）：新建内容（跳转 `/projects/[id]/contents/new`）

#### 4.6.4 权限
- 项目内所有成员可见所有内容（不区分权限，简化 MVP）
- 编辑/删除仅限作者本人
- 项目 owner 不可转让（MVP 不做）

#### 4.6.5 状态
| 状态 | 展示 |
|:--|:--|
| 无项目 | 项目列表页："还没有项目，创建第一个吧" + 创建按钮 + 加入项目入口 |
| 加载中 | Skeleton 卡片 |
| 加载失败 | 错误提示 + 重试 |

## 5. 用户角色与权限

MVP 只有两个角色：ADMIN 和 EDITOR。审核功能由 EDITOR 兼任（任何人可审核非本人提交的内容）。

| 权限 | ADMIN | EDITOR |
|:--|:--|:--|
| 注册/登录 | ✅ | ✅ |
| 创建项目 | ✅ | ✅ |
| 加入项目 | ✅ | ✅ |
| 创建/编辑/删除模板 | 仅限自己的 | 仅限自己的 |
| 创建内容（生成） | ✅ | ✅ |
| 编辑/删除自己的内容 | ✅ | ✅ |
| 提交审核 | ✅ | ✅ |
| 审核他人内容（通过/驳回） | ✅ | ✅ |
| 发布自己的已通过内容 | ✅ | ✅ |
| 查看项目内所有内容 | ✅ | ✅ |

## 7. 数据模型（对应 Prisma Schema）

```
User
  id, email, password, name, role(ADMIN/EDITOR), createdAt

Project
  id, name, description, createdAt
  关系: User(owner) 1:N Project

ProjectMember
  id, role(ADMIN/EDITOR), userId, projectId
  关系: Project N:N User（多对多，一个用户可以加入多个项目）

Template
  id, name, category, promptTemplate(文本，含 {{变量}}), variables(JSON数组),
  projectId, createdAt
  关系: Project 1:N Template

Content
  id, title, body, status(DRAFT/PENDING/APPROVED/REJECTED/PUBLISHED),
  rejectionReason?, templateId?, projectId, authorId, reviewerId?,
  createdAt, updatedAt
  关系: Project 1:N Content, User(author) 1:N Content, Template 1:N Content
```

**关系图**：

```
User ──(owner)──→ Project
  │                  │
  │                  ├──→ Template
  │                  │
  │                  └──→ Content ←── User(author)
  │                                      ←── User(reviewer)
  └──(member)──→ ProjectMember ←── Project
```

## 8. 用户操作流程

### 8.1 主流程：从模板到发布

```
1. 登录 → 进入项目
2. 点「新建内容」→ 选一个模板（比如"小红书种草"）
3. 弹出表单：产品名=防晒霜，目标人群=大学生，风格=活泼 → 点「生成」
4. 看到 AI 一个字一个字打出来（流式输出）
5. 不满意？点「重新生成」或手动编辑
6. 满意了 → 点「提交审核」
7. 状态变"待审核"
8. 切换到审核视角 → 看内容 → 点「通过」（或驳回写理由）
9. 状态变"已通过" → 点「发布」
10. 状态变"已发布"，完。
```

### 8.2 新用户首次使用

```
1. 打开网站 → 重定向到 /login → 点「注册」
2. 填邮箱+密码+姓名 → 提交 → 自动登录
3. 进入 /dashboard（空的："还没有内容"）→ 点「创建项目」
4. 填项目名 → 项目创建成功 → 进入项目详情页
5. 点「模板」Tab → "还没有模板" → 创建第一个模板
6. 点 FAB「新建内容」→ 选模板 → 填变量 → 生成 → 保存草稿
```

### 8.3 多项目切换

```
1. 用户在工作 → 点顶部导航的「项目」→ 看到项目列表
2. 点另一个项目 → 进入该项目详情页
3. 当前项目上下文切换，所有列表只显示该项目数据
```

## 9. 技术决策（为什么这么选）

| 决策 | 选什么 | 为什么 |
|:--|:--|:--|
| 前端 | Next.js 16 | SSR 友好、App Router、React 19 |
| API | oRPC | 端到端类型安全 + 自动 OpenAPI 文档 |
| ORM | Prisma | 类型安全、迁移工具、和 oRPC 配合好 |
| 数据库 | PostgreSQL | 关系型数据天然适合 |
| AI SDK | 直接调 DeepSeek API（OpenAI 兼容） | MVP 不引入抽象层 |
| 流式 | SSE（oRPC Event Iterator） | 轻量、oRPC 原生支持 |
| UI | shadcn/ui + Tailwind | 组件可定制、不臃肿 |
| 认证 | NextAuth.js | 开箱即用、多 provider |

## 10. 页面规格

### 10.1 登录/注册 — `/login`

| 项目 | 内容 |
|:--|:--|
| 用途 | 用户登录或注册 |
| 布局 | 居中卡片，左侧品牌区（logo + 一句话），右侧表单区 |
| 登录模式 | 邮箱输入框 + 密码输入框 + 「登录」按钮 + 「还没有账号？注册」链接 |
| 注册模式 | 邮箱 + 密码 + 确认密码 + 姓名 + 「注册」按钮 + 「已有账号？登录」链接 |
| 表单校验 | 邮箱格式、密码 ≥8 位、确认密码一致、姓名非空 |
| 错误处理 | 密码错误 → 表单下方红色提示"邮箱或密码错误"；邮箱已注册 → "该邮箱已被注册" |
| 登录成功 | 跳转 `/dashboard`（若有 callbackUrl 则跳到 callbackUrl） |

### 10.2 仪表盘 — `/dashboard`

| 项目 | 内容 |
|:--|:--|
| 用途 | 登录后着陆页，跨项目总览 |
| 数据来源 | 全平台（不限项目），按当前用户过滤 |
| 展示 | 见 4.5.1 |
| 空状态 | "还没有内容，去创建第一篇吧" → 按钮跳转到最近的项目或创建项目页 |

### 10.3 项目列表 — `/projects`

| 项目 | 内容 |
|:--|:--|
| 用途 | 查看所有已加入的项目 |
| 展示 | 卡片网格（项目名 + 描述摘要 + 成员数），每行 3 张 |

## 12. 认证实现要点

### NextAuth.js 配置

| 决策 | 选择 | 原因 |
|:--|:--|:--|
| 版本 | NextAuth.js v4（`next-auth@4`） | 稳定、文档全、Prisma 适配成熟 |
| Provider | Credentials Provider（邮箱+密码） | MVP 不引入 OAuth，够用 |
| Session 策略 | JWT | 不存数据库 session 表，减轻 DB 压力 |
| Adapter | `@auth/prisma-adapter` | 直接操作现有 User 表 |

### 环境变量

| 变量 | 说明 |
|:--|:--|
| `NEXTAUTH_URL` | 应用 URL，开发环境 `http://localhost:3001` |
| `NEXTAUTH_SECRET` | JWT 签名密钥，`openssl rand -base64 32` 生成 |

### Middleware 路由保护

```
/login          → 已登录用户重定向到 /dashboard
/dashboard/*    → 未登录重定向到 /login（附 callbackUrl）
/projects/*     → 未登录重定向到 /login
/api/auth/*     → 放行（NextAuth 自身路由）
/api/rpc/*      → 通过 context 注入 session，不强制重定向
```

### 注册流程

1. 用户访问 `/login` → 点「注册」
2. 输入邮箱 + 密码（≥8位）+ 姓名
3. 后端：bcrypt 哈希 → Prisma 创建 User（默认 EDITOR）
4. 注册成功自动登录 → 跳转 `/dashboard`

### API Context 集成

oRPC 的 `createContext` 从 NextAuth 读取 session：

```
createContext(req) → getServerSession() → { session }
                                                     │
                    User ID + role 注入所有 API handler
                    (做权限判断时不需要再查一次用户)
```
